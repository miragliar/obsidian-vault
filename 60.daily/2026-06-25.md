# 2026-06-25 — Donnerstag

> Fortsetzung der Koster-Flow-02-Analyse von [[60.daily/2026-06-24|gestern]]. Heute **kein Designer-Bauen**, sondern zwei Erkenntnisse zementiert: (1) der **KI-Prompt-Screenshot bestätigt die Root Cause direkt** — die KI wird im Output-Format *aufgefordert*, die Dataverse-GUID selbst auszuschreiben → sie halluziniert sie. (2) Die **„braucht es den globalen Catch noch?"-Frage** geklärt → Prinzip **„genau ein Schreiber pro Ausgang"**. Beide Befunde sind in die Tiefen-Notiz von gestern eingeflossen (Abschnitte A/B/C/D + Catch-Kaskade).

## 🔴 Der KI-Prompt bestätigt die Root Cause (Screenshot)

Raoul hat den AI-Builder-Prompt `Run_a_prompt_-_Search_Subunternehmer_1` (Modell **GPT-4.1 mini**) geteilt. Damit ist die gestrige Vermutung **schwarz auf weiss belegt**:

- Der Prompt bekommt die Subunternehmer-Tabelle als Grounding und soll **fuzzy** den passenden Sub finden (Suffixe `-1/-2`, Umlaute, Gross-/Kleinschreibung ignorieren) — das ist ein **sinnvoller LLM-Job**.
- Im geforderten **Output-Format** steht aber `"SubunternehmerID": "GUID"` → die KI soll die **32-stellige Dataverse-Primärschlüssel-GUID verbatim selbst ausschreiben.** **Das** ist der Konstruktionsfehler.
- Merksatz: **Fuzzy-Matching kann ein LLM gut — eine GUID fehlerfrei abschreiben nicht.** (Beleg von gestern: `adresseID` über alle 9 Läufe stabil `41400`, nur die GUID-Suffixe variierten → 404.)

→ **Fix A** steht fest: GUID aus dem Output streichen, nur `Name` + `adresseID` zurückgeben (`{ "result": [ { "Name": "…", "adresseID": "…" } ] }`, „no match" = `{ "result": [] }`). Fuzzy-Matching bleibt.

## 🟣 Braucht es den globalen Catch überhaupt noch?

Raouls Einwand: *„Wenn die Aktionen, die fehlschlagen können, nun richtig reagieren — wozu dann noch ein globaler Catch, der eh alles überschreibt?"* — berechtigt, **solange die Failure-Pfad-Catches drinbleiben.**

Die Auflösung: **Nicht jeder Status-Schreiber trippt den grossen `Scope`.** Die Updates zerfallen in zwei Klassen:

| Klasse | Beispiele | trippt grossen Scope? |
|---|---|---|
| **Erfolgs-Pfad** (Verarbeitet / bewusst Manuell) | `verarbeitet 4`, `verarbeitet 3 3`, `verarbeitet 5`, `verarbeitet 6` | **nein** — `runAfter Succeeded` bzw. Condition-Zweig |
| **Fehler-Pfad** (lokaler Catch nach `Failed`) | `verarbeitet 3 4`, MUSS-2 `Catch - Ablegen fehlgeschlagen` | **ja** — eine `Failed`-Aktion im Scope bleibt `Failed`, auch wenn ein Geschwister-Catch sauber läuft |

Sobald **Fix B** den „nicht gefunden"-Fall als **erwartetes leeres Ergebnis** (Condition auf leere Liste) statt als **Fehler** behandelt, failt der Normalpfad gar nicht mehr → **weder lokaler noch globaler Catch feuern im Normalbetrieb.**

## 🟠 Das eigentliche Überschreib-Problem — „genau ein Schreiber pro Ausgang"

Raouls Kern-Einwand: *„Der globale Catch läuft ja trotzdem, auch wenn ein lokaler schon einen Status gesetzt hat — er überschreibt, das ist das Problem."* — **Genau richtig.** Beide Catches schreiben auf **dieselbe** Zeile, der globale läuft zuletzt → **last writer wins** → die Zeile lügt (gestern: hartkodiert „KI hat Structured Output nicht mitgeliefert").

**Lösung = ein einziger Schreiber pro Ausgang:**
- Die **Fehler-Pfad-Catches `verarbeitet 3 4` und MUSS-2 `Catch - Ablegen fehlgeschlagen` raus** (sie sind die einzigen, die mit dem globalen Catch kollidieren).
- Der **globale Catch bleibt** — aber als **einziger** Last-Resort für *unerwartete* Technikfehler:
  - Status `Manuell (124080003)` → **`Fehlgeschlagen (124080002)`**.
  - Fehlertext **nicht mehr hartkodiert** → **dynamisch** via `result('Scope')` (die real gescheiterte Aktion).
  - `Manuell` ist künftig **ausschliesslich** der legitime „KI konnte nicht zuordnen"-Fall aus Fix B.

Damit kollidiert nichts mehr: Erfolgs-Pfade schreiben ihren Status, der globale Catch fängt nur noch das ab, was sonst niemand abfängt — und überschreibt dabei nichts Korrektes.
