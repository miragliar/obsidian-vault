### Die Kernursache: der KI-Prompt fordert die GUID an — und die KI erfindet sie

Der AI-Builder-Prompt `Run_a_prompt_-_Search_Subunternehmer_1` (Modell **GPT-4.1 mini**) bekommt die Subunternehmer-Tabelle als Grounding (`Name`, `adresseID`, `Subunternehmer`, `adrName`) und die Anweisung *„Return 1 item from the Subunternehmer table where adrName or Name (fuzzy) matches … Return the best match."* — also ein reiner **Fuzzy-Match-Job** (Suffixe/Umlaute/Case ignorieren). Im geforderten **Output-Format steht aber zusätzlich `"SubunternehmerID": "GUID"`** → die KI soll die Dataverse-Primärschlüssel-GUID **selbst ausschreiben.** Das ist der Konstruktionsfehler: **Fuzzy-Matching kann ein LLM gut, eine 32-stellige Hex-GUID verbatim kopieren nicht.** (Beleg: `adresseID` blieb über alle 9 Läufe stabil `41400` — nur die GUID variierte.)

`Get_a_row_by_ID` bekommt seinen Schlüssel direkt aus diesem KI-Output:
```
recordId = @outputs('Run_a_prompt_-_Search_Subunternehmer_1')
            ?['body']?['responsev2']?['predictionOutput']
            ?['structuredOutput']?['result']?[0]?['SubunternehmerID']
```
Die KI soll also die **Dataverse-Primärschlüssel-GUID** des Subunternehmers ausschreiben — und **erfindet** sie. Gleicher Sub „Die Bau Leu GmbH", aber jeder Lauf eine andere GUID:

| Quelle | SubunternehmerID | Vorkommen (Run CU18, 9 Dok., alle „Die Bau Leu GmbH") |
|---|---|---|
| **Dataverse (echt)** | `43cfe1d8-aa34-f111-88b4-002248802`**`f9f`** | existiert ✅ |
| KI erfunden A | `43cfe1d8-aa34-f111-88b4-002248802`**`b4f`** | 6/9 → 404 |
| KI erfunden B | `43cfe1d8-aa34-f111-88b`**`5-000d3a2a7fc3`** | 3/9 → 404 |

→ Gleicher Prefix, fabrizierter Suffix = klassische LLM-Halluzination. „Get a row by ID" auf eine nicht existierende GUID → **404** → `Scope_-_keine_deklaration_id_aber_Sub_Name_1` failt → globaler Catch stempelt fälschlich „KI hat Structured Output…" + Manuell.

**„Die Bau Leu GmbH" existiert in Dataverse** (genau der Sub, dessen SUVA du manuell abgelegt hast). Also **kein „unbekannter Subunternehmer"**, sondern ein **bekannter Sub, dessen Schlüssel die KI nicht zuverlässig liefern kann.**

**Lage im Flow:**
`For_each → Condition_-_PNG_Filter → Scope_-_Dokument_verarbeiten → Apply_to_each → Scope → Condition_-_hat_Deklaration_ID_1 [ELSE] → Condition_-_keine_Deklaration_ID_aber_Sub_Name_1 → Scope_-_keine_deklaration_id_aber_Sub_Name_1 → Get_a_row_by_ID_-_Subunternehmer_Infos_holen_1`

## 🟣 Warum die Zeile FALSCH beschriftet wird (Catch-Kaskade)

Offene Frage war: Wenn der lokale Catch `verarbeitet_3_4` erfolgreich lief — warum greift der globale Catch trotzdem? **Antwort: ein erfolgreicher Catch heilt den Scope nicht.** Ein Scope/Condition/Loop bekommt seinen Status aus der **Aggregation seiner Kinder** — eine einzige `Failed`-Aktion darin macht den ganzen Container `Failed`; ein erfolgreicher Geschwister-Catch (`runAfter Failed`) lässt den Flow nur **weiterlaufen**, nimmt den Fehlerstatus aber **nicht** zurück.

Zwei Catches auf zwei Ebenen:
- **lokal** `…verarbeitet_3_4` → `runAfter {Scope_-_keine_deklaration_id_aber_Sub_Name_1: [Failed]}` (kleiner Scope)
- **global** `…Verarbeiten_fehlgeschlagen_-_aber_noch_manuell` → `runAfter {Scope: [Failed]}` (grosser `Scope`, direkt unter `Apply_to_each`)

Der 404 „blubbert" Ebene für Ebene hoch (Status real aus der Run-History, CU18):

| Ebene | Status |
|---|---|
| `Get_a_row_by_ID` | ❌ Failed (404) |
| `Scope_-_keine_deklaration_id_aber_Sub_Name_1` (klein) | ❌ Failed |
| `verarbeitet_3_4` (lokaler Catch) | ✅ Succeeded → schreibt korrekt „Fehler bei Condition – keine Deklaration ID aber Sub Name" |
| `Condition_-_keine_Deklaration_ID_aber_Sub_Name_1` | ❌ Failed — *trotz* erfolgreichem Catch darin |
| `Condition_-_hat_Deklaration_ID_1` | ❌ Failed |
| `Scope` (gross) | ❌ Failed |
| globaler Catch (`runAfter Scope=Failed`) | ✅ Succeeded → **feuert** |

Beide Catches schreiben auf **dieselbe** Zeile (`recordId = Add_a_new_row_1 → ks_eingangsqueueid`): lokal zuerst (korrekter Text), global danach → **überschreibt** mit dem **hartkodierten** String „KI hat Structured Output nicht mitgeliefert" + Manuell. **Last writer wins** → die Zeile lügt. Der Text ist eine Konstante im globalen Catch, kein echter KI-Fehler — deshalb sah es anfangs wie ein KI-Output-Problem aus.

## 🟠 Nächste Schritte — der eigentliche Fix (noch offen)

### A) KI-Prompt — die GUID rauswerfen
Die KI darf die GUID **nicht** mehr ausgeben (sie kann sie nicht verbatim kopieren). Sie soll nur den **kanonischen Namen** (+ `adresseID`) zurückgeben — exakt der Tabellenwert, den sie ohnehin matcht.
- Im Output-Format `"SubunternehmerID": "GUID"` **streichen** → `{ "result": [ { "Name": "company name", "adresseID": "company ID" } ] }`; „no match" weiter `{ "result": [] }`.
- Zusatz im Prompt: *„Return Name and adresseID exactly as they appear in the table."*
- Das **Fuzzy-Matching bleibt** — das ist die Stärke des LLM und der einzige Grund, hier überhaupt KI zu nutzen.

### B) Flow — Lookup deterministisch machen
1. `Get_a_row_by_ID_-_Subunternehmer_Infos_holen_1` **ersetzen** durch **„Zeilen auflisten"** auf `crb4b_subunternehmers`, Filter `crb4b_name eq '<KI-Name>'` (Top 1) → echte GUID = `first(outputs(...)?['body/value'])?['crb4b_subunternehmerid']`.
2. **Auch** `List_rows_-_Auftrag_holen_1` umstellen: filtert bisher `_ks_lookupsubunternehmer_value eq '<KI-GUID>'` → muss die **echte** GUID aus Schritt 1 nehmen, sonst läuft auch dieser Filter ins Leere.
3. **Kein Treffer (leere Liste) → bewusst `Manuell` „Subunternehmer nicht erkannt"** über eine **Condition auf das leere Ergebnis**, NICHT über einen Fehler. Effekt: Der Scope failt auf dem Normalpfad nicht mehr → **der globale Catch feuert nicht mehr → kein Überschreiben.** (Das ist der eigentliche Hebel gegen die Catch-Kaskade oben.)

### C) Globalen Catch entschärfen
`Update_a_row_-_Verarbeiten_fehlgeschlagen_-_aber_noch_manuell`:
- Fehlertext **nicht mehr hartkodiert** „KI hat Structured Output…" → dynamisch/generisch („Technischer Fehler bei der Verarbeitung").
- Status `Manuell (124080003)` → **`Fehlgeschlagen (124080002)`**. `Manuell` künftig **ausschliesslich** für den legitimen „KI konnte nicht zuordnen"-Fall aus B3.
- Damit ist er ein echter Last-Resort für *unerwartete* Technikfehler und kollidiert nicht mehr mit dem lokalen Catch.

### D) Optional
Denselben Sub-Lookup im MA-Zuweisungs-Flow (App-Seite) prüfen, falls dort auch KI-GUIDs verwendet werden.

**Reihenfolge:** A → B → C. Nach B feuern lokaler/globaler Catch im Normalfall gar nicht mehr, weil „nicht gefunden" dann kein *Fehler* mehr ist, sondern ein erwartetes leeres Ergebnis.
